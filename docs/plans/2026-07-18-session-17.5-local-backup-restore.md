# Session 17.5 — Free Local JSON Backup & Restore

## Context

PLAN-2.md Session 17.5 is now the **sole backup path** for Sugar (auth + cloud backup dropped 2026-07-18). Goal: any user — free, signed-out, any mode — can get all their data off the phone into a versioned JSON file (via the OS share sheet → Drive/iCloud/Files/Zalo/email) and restore it onto a new device. This is the "free tier is sacred / you can always take your data with you" guarantee and the migration safety net.

**Design is already fully specified** in `design/Sugar App.dc.html` (source of truth):
- A chevron **nav row** in Settings → "Dữ liệu" section, showing the last-backup date (lines 518–523).
- A dedicated **Data backup screen** (lines 869–898): intro blurb + card with "Export backup file" (green `upload_file`) and "Restore from file" (purple `download`) rows + "Last backup: {date}" footnote.
- A **restore-confirm bottom sheet** (lines 900–911): icon tile, "Restore this backup?", warning body, primary Restore + ghost Cancel.

**User decisions:** dedicated `/backup` screen (per design, not inline section) + `BottomSheet` restore confirm (per design, not `Alert`).

**Never gated** (money-principle #1) — do NOT wire through `useProGate`.

## Backup format (canonical, versioned)

```ts
interface BackupFile {
  app: 'sugar';
  schemaVersion: number;   // backup-format version WE own; start at 1; bump on shape change. Independent of drizzle migration ids.
  exportedAt: number;      // Unix ms
  readings: Reading[];     // domain models (NOT the SQLite row layout)
  settings: AppSettings;   // domain settings object
}
```
Serialize **domain models** so DB migrations never break old files.

## Implementation

### 1. Dependency
- `npx expo install expo-document-picker` (needed for restore; `expo-file-system` ~19 and `expo-sharing` ~14 already installed).

### 2. Domain layer (pure TS, unit-tested — the seam)
New file `src/domain/models/backup.ts`: `BackupFile` interface + `BACKUP_SCHEMA_VERSION = 1` const.

New use-cases in `src/domain/use-cases/` (mirror the pure style of `export-readings-csv.ts`):
- `serialize-backup.ts` — `serializeBackup(readings: Reading[], settings: AppSettings, now: number): BackupFile`.
- `parse-backup.ts` — `parseBackup(input: unknown): BackupFile`. Validate `app === 'sugar'`, narrow every field with type guards (never `any`, use `unknown`). **Reject** a file whose `schemaVersion > BACKUP_SCHEMA_VERSION` (throw a typed error → screen shows "Tệp này từ phiên bản Sugar mới hơn — hãy cập nhật app"). Throw a distinct typed error for malformed/wrong-app files.
- `migrate-backup.ts` — `migrateBackup(file: BackupFile): BackupFile`. Forward-migrate older `schemaVersion` to current. v1 is identity; the switch scaffold exists for future bumps.
- `apply-backup.ts` — `applyBackup(file, deps): Promise<{ restored: number; skipped: number }>`. **Replace** semantics: re-validate each reading (reuse `isWithinNormalRange` from `validate-reading-value.ts` — 20–600 warn-only preserved; skip + count garbage rows, never abort), then `readingRepo.replaceAll(goodReadings)` (atomic) + `settingsRepo.clear()` then persist each `file.settings` key. **Preserve original `id`/`createdAt`** — do NOT route through `createReading` (it regenerates them).

Use error subclasses (e.g. `BackupTooNewError`, `BackupMalformedError`) so the UI can branch on `instanceof` for the right i18n message.

### 3. Repository port + adapter (bulk atomic insert)
Add to `src/domain/repositories/reading-repository.ts`:
```ts
/** Atomically replace every reading (clear + bulk insert) — used by restore. */
replaceAll(readings: Reading[]): Promise<void>;
```
Implement in `src/data/repositories/sqlite-reading-repository.ts` using `this.db.transaction(tx => { tx.delete(readings).run(); if (rows.length) tx.insert(readings).values(rows).run(); })`, reusing the existing `toRow` mapper (undefined→null). This is the first use of `db.transaction` — the sync drizzle driver supports it. Also update the in-memory test double if one exists.

### 4. Data layer side-effects — `src/data/backup/`
- `share-backup.ts` — `exportBackup(deps): Promise<ExportBackupResult>` mirroring `share-csv.ts`: `serializeBackup` → `JSON.stringify` → `new File(Paths.cache, 'sugar-backup-YYYY-MM-DD.json')` → `file.create({overwrite:true})` + `file.write` (UTF-8 plain text; the UTF-16 BOM trick is CSV/Excel-only, not needed for JSON) → `Sharing.isAvailableAsync()` → `Sharing.shareAsync(uri, { mimeType: 'application/json', UTI: 'public.json', dialogTitle })`. Status enum `{ Shared, Unavailable }` (0 readings is still a valid backup — no Empty).
- `import-backup.ts` — `pickBackupFile(): Promise<string | undefined>`: `DocumentPicker.getDocumentAsync({ type: 'application/json' })` → on cancel return `undefined` → else `new File(uri).text()`. Parsing/migrating/applying is done by the domain use-cases from the caller (keeps this module pure I/O).

### 5. Settings key + store re-hydration
- `src/domain/models/settings.ts`: add `lastLocalBackupAt: number | null` to `AppSettings` + `DEFAULT_SETTINGS` (default `null`). No migration (schema-less JSON kv store).
- `src/ui/hooks/use-settings.ts`: add `lastLocalBackupAt` to the parallel `initialize()` read list + `set()` block. Write it after a successful export via existing `updateSetting('lastLocalBackupAt', now)` (exactly how `report.tsx` bumps `reportCount`).
- **Re-hydration after restore:** `initialize()` early-returns on `isInitialized`, so restored settings rows won't reach the in-memory store. Add a `rehydrate(): Promise<void>` action that re-reads all keys unconditionally (refactor: extract the read+set body of `initialize` into a shared internal `load()`; `initialize` guards then calls it, `rehydrate` calls it directly). Call `rehydrate()` after `applyBackup`.

### 6. UI
- **Settings nav row** — `app/(tabs)/settings/index.tsx`, in the "Dữ liệu" `Card` (after Doctor report, before About): a `SettingRow` with `icon="cloud-upload"` (or `save`), `iconColor={colors.primary}`, `label` = `rows.backup`, `value` = formatted `lastLocalBackupAt` (or a "Chưa sao lưu" placeholder), `onPress={() => router.push('/backup')}`.
- **New screen** `app/backup.tsx`: intro text (design blurb), a `Card` with two `SettingRow`s — Export (`icon="cloud-upload"`, `colors.primary`) and Restore (`icon="cloud-download"`, `colors.accentPurple`) — and a muted "Lần sao lưu gần nhất: {date}" footnote.
  - **Export:** call `exportBackup`; on `Shared` → `updateSetting('lastLocalBackupAt', Date.now())` + `analytics.backupEnabled()` + success toast/Alert; on `Unavailable` → Alert. `isBusy` state + `haptics`.
  - **Restore:** `pickBackupFile()` → if undefined, no-op (cancelled) → `parseBackup`/`migrateBackup` (catch `BackupTooNewError` / `BackupMalformedError` → specific Alert) → open `BottomSheet` confirm ("Khôi phục bản sao lưu này?" + warning) → on confirm: `applyBackup` → `rehydrate()` settings + refresh readings (see note) → success Alert "Đã khôi phục {n} chỉ số" (append skipped-row note if `skipped > 0`).
- **Register** in `app/_layout.tsx` root `<Stack>`: `<Stack.Screen name="backup" options={{ headerShown: true, title: t('screens.settings.backup.title') }} />`. Native back chevron is automatic (no manual `headerLeft`).
- **Readings refresh after restore:** confirm how `src/ui/hooks/use-readings.ts` refetches (focus-based vs store). If it reloads on tab focus, navigating back suffices; otherwise trigger its refresh action after `rehydrate()`.
